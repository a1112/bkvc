Visionary-B Two Camera Features Overview

This document serves as a preliminary overview of the GenICam features provided by the Visionary-B Two camera.

In case of any doubts, contact us for further information.

GenICam Interface Overview

GigE Vision cameras expose their functionality through GenICam standard compliant feature description. Wherever possible, the Visionary-B Two strictly follows the GenICam Standard Features Naming Convention (SFNC).

The current version of the SFNC document can be downloaded from the official website for further reference where needed.

In most cases, however, the brief description in this document should be fully sufficient and checking the SFNC document might mainly be useful to understand the GenICam 3D features model.

Note that features might have limited access mode (readability/writability) and some might become read-only or even not-available depending on other features or current state of the camera. In particular, many of the features cannot be changed during active acquisition. Numeric features will always report their current limits.

The GenICam SFNC document sorts features into "categories", which are mainly useful when viewing the features in a GUI application. The categories are not relevant when accessing (reading/writing) the features programmatically. For clarity, we nevertheless sort the features according to their categories in following chapters.

GenICam Feature Categories

DeviceControl

Mostly read-only information about the device.

DeviceScanType

Currently always "Areascan3D"

DeviceVendorName

Vendor name ("SICK")

DeviceModelName

Model name ("Visionary-B Two")

DeviceVersion

Firmware version of the device as a string

DeviceSerialNumber

Serial number of the device as a string

DeviceLinkHeartbeatMode

Debug feature, do not use without good reason

DeviceLinkHeartbeatTimeout

Debug feature, do not use without good reason

DeviceOperationStatus

Reports current operation status of the device

Ready

Device is in running state, ready for operation

Warning

Device is in running state, but with a warning asserted (for example temperature exceeds warning level)

Error

Device is in running state, but with an error reported (reliable operation not guaranteed)

NotReady

Device is not ready for operation (for example in transition between states)

DeviceReset

Resets the device to its power up state (performing a reboot). After reset, the device must be rediscovered

ImageFormatControl

Features controlling the image components to be acquired and their parameters.

ComponentSelector

Selects among supported image data components to be controlled through depending features. Supported components are

Range

The range (distance) data of individual pixels

Intensity

The intensity (color) data of individual pixels

ImuBasic

The (max.) last 100 IMU data samples

ComponentEnable

Boolean feature enabling/disabling acquisition of the component currently selected in ComponentSelector. Note that to be able to enable more than one component at once, the connected GigE Vision receiver must support the GigE Vision "multi-part" streaming mode (introduced in GigE Vision version 2.1). Otherwise enabling multiple components will not be possible - to switch between them in such case, first disable the current, then enable the new component.

Width

Width (in pixels) of the component currently selected in ComponentSelector. Currently fixed (read-only).

Height

Height (in pixels) of the component currently selected in ComponentSelector. Currently fixed (read-only).

PixelFormat

Format of the pixels of the component currently selected in ComponentSelector. Currently fixed (read-only) for every component type:

Coord3D_C16

For range component, 16-bit integer pixel values.

BGR8

For intensity component, blue-green-red 8-bit per channel pixel values.

Mono8

For ImuBasic data raw buffer, for decoding please see IMU data related examples in both Python and C++ example sets of this package.

AcquisitionControl

AcquisitionMode

Currently always "Continuous".

AcquisitionStart

Starts acquisition of the device (if GigE Vision stream channel is open). Most GigE Vision receivers handle this feature under the hood, wrapping it in its own acquisition control APIs. In such case the feature should not be touched manually.

AcquisitionStop

Stops acquisition of the device. Most GigE Vision receivers handle this feature under the hood, wrapping it in its own acquisition control APIs. In such case the feature should not be touched manually.

AcquisitionFrameRate

Allows to reduce the acquisition frame rate from its maximal value, in Hz. Available only if ExposureAuto is set to Off.

ExposureTime

Allows to control the exposure time (in microseconds) if ExposureAuto is Off. If ExposureAuto is set to Continuous, the feature returns the exposure time currently used by the ExposureAuto algorithm.

ExposureAuto

Controls the automatic exposure mode. Depending on the selected state, access to other features may change (see below). Possible states are:

Off

Automatic exposure not used, exposure time controlled through the ExposureTime feature manually. Blocks access to: ExposureAutoFrameRateMin.

Continuous

Automatic exposure mode used by the camera. Blocks access to: AcquisitionFrameRate. Write protects ExposureTime.
Note: this mode needs a few frames to adapt to changes in the scene.

ExposureAutoFrameRateMin

Sets the minimal required frame rate (in Hz) when ExposureAuto is Continuous. Technically applies upper limit to the automatic exposure time to meet the required frame rate. Available only if ExposureAuto is set to Continuous.

MultiSlopeMode

Controls multi-slope exposure mode, allowing to improve the acquired data dynamic range. Possible values are:

Off

Standard mode, single exposure. Reduce motion artifacts.

PresetAggressive

High dynamic range (HDR) mode, using two separate exposures. Suitable to cover both dark and shiny objects. But might increase motion artifacts and lower maximal possible frame rate. Only recommended for static scenes.

Scan3dControl

Features controlling the 3D data related features.

FieldOfView

Switches among different field of view angles. For specific values see the operating manual of your Visionary device.

Wide

Capture a wider field of view; more shrinking/compression due to fisheye distortion.

Narrow

Capture a narrower field of view; more details in the center region and higher accuracy over longer distances.

Scan3dDataFilterSelector

Selects among supported data filters to be controlled (enabled/disabled) on the camera. Currently the only supported filter is

ValidationFilter

Invalidates pixels with validity confidence under selected threshold (Scan3dDepthValidationFilterLevel).

Scan3dDataFilterEnable

Boolean feature enabling/disabling the data filter currently selected in Scan3dDataFilterSelector.

Scan3dDepthValidationFilterLevel

Sets the required level of pixel confidence used by the validation filter. Value meaning: -9 = weakest, 0 = balanced, 9 = strongest filter. Weaker filter means more "valid" points in the output, possibly including noise/artifacts. Stronger filter means the uncertain points are removed from the output, sometimes possibly including actually some valid measurements.

ChunkDataControl

Contains features controlling reception of the stream metadata and metadata itself. The stream metadata are called "chunk data" in GenICam world.

First the features allowing to switch chunk data delivery together with the data stream (range and/or intensity images). It is recommended to always enable the chunk data together with the range component, because they carry information required to build the point cloud from the depth map.

ChunkModeActive

Boolean feature enabling or disabling chunk delivery together with the (image) data stream. It is recommended to always enable and use the chunk data to properly interpret the acquired depth (range) map.

ChunkSelector

Present mainly for GenICam SFNC compatibility, listing all the chunk data supported by the camera. Enabling them individually is not, however, supported, the ChunkModeActive feature enables/disables all of them at once.

ChunkEnable

Present mainly for GenICam SFNC compatibility, Boolean feature representing whether the chunk feature selected in ChunkSelector is currently enabled. Always "true" on this camera, the ChunkModeActive feature enables/disables all of them at once.

Next the individual chunk features carrying the information about the camera’s 3D model (incl. camera intrinsic parameters) require to build the point cloud. If enabled (through ChunkModeActive), the data will be transmitted together with the data stream for each block ("buffer").

GenICam standard provides a mechanism to parse the delivered chunk data and allow the user to read them as any other camera features. Most GigE Vision receivers will perform these operations for you under the hood and chunk features can be read (same as other features) after each acquired "image". Note that before the first acquisition (or if chunks were not enabled), the following features will not be available.

The following chunk features adhere to the 3D model described by the GenICan SFNC standard - we recommend to refer to that document for detailed understanding.

The "ProjectedC" output mode, is, however, still a pending proposal for SFNC extension. To keep the explanation simple, the algorithm to compute the point cloud from the acquired depth map is provided in PointCloudGeneration.pptx, also delivered with the beta version devices.

Given that, following is the list of the chunk data features generated by the camera, for their use refer to above mentioned document(s):

  • ChunkScan3dOutputMode

  • ChunkScan3dCoordinateSelector

  • ChunkScan3dCoordinateScale

  • ChunkScan3dCoordinateOffset

  • ChunkScan3dInvalidDataFlag

  • ChunkScan3dInvalidDataValue

  • ChunkScan3dFocalLength

  • ChunkScan3dAspectRatio

  • ChunkScan3dPrincipalPointU

  • ChunkScan3dPrincipalPointV

  • ChunkScan3dDistanceUnit

  • ChunkScan3dCoordinateSystem

Following chunk features are also supported by the camera, but have limited use in current firmware version, please ignore them:

  • ChunkScan3dCoordinateSystemReference

  • ChunkScan3dCoordinateReferenceSelector

  • ChunkScan3dCoordinateReferenceValue

  • ChunkComponentIDValue

Note: some (few) advanced GigE Vision receivers might be able to read those chunk data features and build the point cloud for you under the hood.

EventDataControl

Contains features controlling reception of the asynchronous camera events and event data itself. The events are transported over a dedicated channel (called "message channel") in the GigE Vision protocol.

First the features allowing to switch generation and delivery of individual event types ON and OFF.

EventSelector

Lists all the event types supported by the camera and allows to enable generation & delivery of the selected event using the EventNotification feature. Supported events are:

Test

Intended to allow reliable testing of the events functionality, independent on the actual state of the camera. Required by GigE Vision specification, this test cannot be enabled/disabled using EventNotification, it is always enabled. The dedicated TestEventGenerate command feature allows to generate the event.

OperationStatusChange

The event will be generated when the device operation status changes (new value should be read from DeviceOperationStatus).

EventNotification

Enumeration feature allowing to control (enable/disable) generation of the event type selected in EventSelector. Note that the Test event is always enabled and cannot be switched off. The enumeration options of this feature are:

Off

The event selected in EventSelector is disabled, the camera will not generate any instances of this event type.

On

The event selected in EventSelector is enabled, the camera will send new instances of this event type to the application as they appear. It is up to the application to listen for them and process them.

Next the individual event features allowing to monitor reception of individual event instances and reading data they carry. The SFNC specification defines a convention for naming event related features. For each event type supported by a camera, for example event Foo, there is a subcategory called EventFooData. That subcategory contains feature EventFoo which can be used by the application as a target to listen for GenICam notifications (generated when new event of given type arrives). The subcategory also contains feature EventFooTimestamp carrying device timestamp from the time the event was generated. If the event carries additional data, more features describing that data can be present in the subcategory. The GenIStream SDK tutorial example demonstrates how to use these features.

GenICam standard provides a mechanism to parse the delivered event data and allow the user to read them as any other camera features (event subcategories described above). Most GigE Vision receivers will perform these operations for you under the hood and event features can be read (same as other features) after each received "event". Note that before the first received event of given type (or if events were not enabled), the following features will not be available.

Note for Python users: handling of the device events is currently not implemented reliably in Harvesters.

The following event features adhere to the naming convention described above. The purpose of individual event types is documented in EventSelector (also above).

EventTestData

Sub-category carrying features related to the "Test" event type.

EventTest

Feature intended to be used as a target to register GenICam feature notifications used to monitor received events of the "Test" event type.

EventTestTimestamp

Timestamp of the last delivered event of the "Test" event type.

EventOperationStatusChangeData

Sub-category carrying features related to the "OperationStatusChange" event type.

EventOperationStatusChange

Feature intended to be used as a target to register GenICam feature notifications used to monitor received events of the "OperationStatusChange" event type.

EventOperationStatusChangeTimestamp

Timestamp of the last delivered event of the "OperationStatusChange" event type.

TransportLayerControl

Low level features controlling the "transport layer" (GigE Vision) specific parameters.

PayloadSize

Maximal size of the data to be transferred in single "block" given the current camera configuration. Used by the receiver to allocate stream buffers. Handled under the hood by most GigE Vision receivers.

GigEVision

GigE Vision specific sub-category of TransportLayerControl.

GevMACAddress

MAC address of the device Ethernet interface.

GevCurrentIPAddress

Current (read-only) IP address of the device.

GevCurrentSubnetMask

Current (read-only) subnet mask of the device.

GevCurrentDefaultGateway

Current (read-only) default gateway of the device.

GevCurrentIPConfigurationPersistentIP

Boolean feature controlling whether the device should attempt to use the persistent IP settings (see GevPersistentIPAddress, GevPersistentSubnetMask and GevPersistentDefaultGateway) within the IP configuration procedure upon the next power cycle. If enabled, the persistent IP is the first IP configuration scheme considered by the device.

GevCurrentIPConfigurationDHCP

Boolean feature controlling whether the device should attempt to use the DHCP procedure within the IP configuration procedure upon the next power cycle. If enabled, the DHCP is used only if the persistent IP (GevCurrentIPConfigurationPersistentIP) is disabled or if it fails.

GevCurrentIPConfigurationLLA

Boolean feature controlling whether the device should attempt to use the LLA (link-local address) protocol within the IP configuration procedure upon the next power cycle. Always enabled, but used only as last resort protocol, if both persistent IP and DHCP options are disabled or if they fail.

GevPersistentIPAddress

Persistent IP address of the device - to be applied upon next power cycle if the IP configuration option for persistent IP is enabled (refer to GevCurrentIPConfigurationPersistentIP).

GevPersistentSubnetMask

Persistent subnet mask of the device - to be applied upon next power cycle if the IP configuration option for persistent IP is enabled.

GevPersistentDefaultGateway

Persistent default gateway of the device - to be applied upon next power cycle if the IP configuration option for persistent IP is enabled.

GevSCPSPacketSize

Size of the packets (in bytes) the device will use to stream acquired data (images). Most GigE Vision receivers will auto-negotiate optimal size given the device limits and current MTU of the receiver’s network interface. It is therefore not recommended to write this feature manually unless one knows well what to do (otherwise stream might fail). For good stream performance (no or minimal packet loss), the packet size should be as big as possible. If the auto-negotiated size is not bigger than 1500, check configuration of your network card and enable "jumbo frame" support (typically 9k).

GevSCPD

Allows to control the minimal inter-packet delay to insert among consecutive stream packets (in nanoseconds). Used as elementary flow control mechanism in case of performance problems on receiver side - depending on the stream speed and processing load, the receiver’s network stack might be dropping some of the incoming stream packets. Increasing the packet delay (default is zero) will slow down the stream peaks and might help the receiver to cope with the incoming data. Use with care when experiencing performance problems - note that too big inter-packet delay will not allow the camera to transfer all its data. When in doubts, contact us for further advice.
